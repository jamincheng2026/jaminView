# vPro Tech Stack V1

## 1. 文档目标

这份文档用于明确 `vPro` 第一版的前端技术选型，尤其是：

- 页面框架怎么定
- 图表库怎么选
- 国内 3D 地图怎么做
- 表单、状态、编辑器交互分别用什么
- 哪些方案暂时不进入 V1

本文件的定位不是“永久不变的架构真理”，而是当前版本的统一基线。

---

## 2. 当前产品阶段对应的技术原则

当前项目仍处于 `V1 / 产品原型与正式实现前期`，所以技术方案优先遵守以下原则：

1. 先统一，不混搭太多库
2. 先可落地，再追求理论最优
3. 优先适配国内业务和演示场景
4. 优先满足视觉效果、组件封装和后续维护
5. 不为了未来 AI 愿景提前引入复杂技术

---

## 3. 推荐技术栈总览

### 3.1 应用框架

- `Next.js`
- `React`
- `TypeScript`
- `pnpm`

推荐原因：

- 同时满足官网首页、营销页、登录后产品页的统一承载
- 首页和模板页可以做好 SEO
- 编辑器页可以作为 `client-heavy` 页面实现
- 方便后续继续扩展文档页、案例页、模板页、发布展示页

### 3.2 UI 与设计系统

- `shadcn/ui`
- `Radix UI`
- `Tailwind CSS`

推荐原因：

- 适合做自己的视觉系统，而不是被现成后台风格锁死
- 组件基础稳定，可访问性更好
- 对 `Next.js` 友好
- 适合后续建立品牌主题、编辑器主题和组件规范

注意：

- `shadcn/ui` 只作为底层基座
- 不直接沿用默认视觉
- 最终页面风格必须由项目自己的 token 和组件样式控制

### 3.3 表单与校验

- `react-hook-form`
- `zod`

推荐原因：

- 适合配置面板、登录表单、导入配置、属性编辑表单
- 性能和开发体验都比较稳
- `zod` 适合统一表单校验与数据结构验证

### 3.4 图表

- `VChart`

### 3.5 地图

- `高德地图 JSAPI 2.0`
- `Loca`
- `AntV L7`

### 3.6 编辑器交互

- `dnd-kit`
- `react-rnd`

推荐原因：

- 适合拖入、排序、拖拽与缩放
- 第一版可以先完成最基本的画布编辑能力

### 3.7 数据导入

- `xlsx`

推荐原因：

- 支持 `Excel / CSV`
- 配合 JSON 导入可以覆盖第一版静态数据接入需求

## 4. 图表库选型结论

## 4.1 最终建议

`V1 图表主库建议选 VChart。`

原因不是 `ECharts` 不行，而是基于当前项目阶段，`VChart` 更符合以下优先级：

- 更现代的默认视觉表现
- 更适合强调质感和风格一致性的产品首页与大屏组件
- React 接入更自然
- 主题系统对“做出更像设计稿的图表”更友好

## 4.2 为什么不是继续默认选 ECharts

`ECharts` 依然是成熟可用的方案，并不算“不能用”。

它的优点：

- 图表类型全
- 生态成熟
- 社区资料很多
- 大屏场景验证充分

但对当前项目来说，它的问题不是能力不够，而是：

- 默认视觉更偏传统
- 如果不自己重做主题，容易有“旧大屏”味道
- 在高 UI 要求下，第一眼气质不一定最符合现在的目标

结论：

- 如果项目目标是“稳、全、熟悉”，`ECharts` 仍然是安全选项
- 如果项目目标是“默认更现代、视觉更好、React 更顺手”，优先 `VChart`

## 4.3 为什么 V1 不建议同时上 VChart 和 ECharts

同时维护两套图表栈，会直接带来这些问题：

- 两套 option/schema 不统一
- 主题系统难统一
- 配置面板难统一
- 导出与保存结构容易发散
- 团队心智成本更高

所以 V1 建议：

- 只定一个主图表栈
- 后续确实遇到 VChart 不好覆盖的少数图表，再局部补充

## 4.4 VChart 在项目里的使用方式

第一版建议这样封装：

- 统一 `ChartContainer`
- 每一种图表类型提供自己的 schema adapter
- 组件层只认项目内部的配置格式，不直接暴露原始图表库复杂结构

推荐组件分类：

- 折线图
- 柱状图
- 面积图
- 饼图 / 环图
- 雷达图
- 漏斗图
- 仪表盘
- 散点图
- 排行榜图表
- 数值卡 / 趋势卡

## 5. 地图方案结论

## 5.1 最终建议

如果第一版面向国内业务，地图能力建议分两层：

### 第一层：基础国内 3D 底图

- `高德地图 JSAPI 2.0`

适合：

- 国内城市地图
- 倾斜视角
- 建筑感底图
- 标准地图交互

### 第二层：地图可视化表达

- `Loca`
- `AntV L7`

适合：

- 点位
- 飞线
- 热力
- 轨迹
- 聚合
- 区域可视化
- 更强的空间数据表达

## 5.2 怎么分工使用

### 路线 A：先快落地

- 底图：`高德 JSAPI`
- 动效表达：`Loca`

适合：

- 第一版快速出国内大屏效果
- 更偏业务演示
- 地图是页面中的一个重要组件，但不是整个产品核心引擎

### 路线 B：地图能力更重

- 底图 / 场景：`高德 + L7`

适合：

- 后续地图会成为产品核心部分
- 要做更多空间可视化图层
- 要追求更强的 2D/3D 数据表现

## 5.3 V1 推荐落地方式

建议先采用：

- `高德 JSAPI 2.0 + Loca`

同时在技术方案中预留：

- `L7` 作为后续增强选项

原因：

- 国内数据和底图体验更稳
- 上手成本相对更可控
- 第一版足够完成“好看且能演示”的 3D 地图组件

## 6. 不同地图场景怎么选

### 场景 1：全国 / 省市区域分布图

推荐：

- `VChart` 或 `ECharts + GeoJSON`

适合：

- 中国地图
- 省份着色
- 城市散点
- 简单飞线

说明：

- 这类“行政区可视化”不一定需要真正地图引擎
- 如果只是区域图，不要默认上高德底图

### 场景 2：城市级三维展示大屏

推荐：

- `高德 JSAPI + Loca`

适合：

- 城市运营
- 门店分布
- 轨迹
- 热力
- 3D 城市场景展示

### 场景 3：后续更强的空间可视化产品

推荐：

- `高德 + L7`

适合：

- 多图层
- 更复杂的交互
- 更明显的 3D 可视化表达

## 7. Next.js 里的技术实现建议

图表和地图组件都建议按 `Client Component` 处理。

原因：

- 图表和地图本质上依赖浏览器环境
- SSR 对这类组件收益有限
- 强行 SSR 反而会增加适配成本

建议策略：

- 官网首页、模板页、文档页：可 SSR / SSG
- 编辑器页：客户端重交互
- 图表组件：动态加载
- 地图组件：动态加载

建议做法：

- 使用 `next/dynamic`
- 图表和地图容器统一封装加载态、空态和错误态
- 不把图表库原始 API 直接暴露到业务页面

## 8. 主题系统建议

图表和地图的视觉效果，不要依赖库默认主题。

第一版就应该建立：

### 8.1 品牌主题

用于官网、产品壳子、基础控件

### 8.2 大屏主题

用于大屏容器、图表卡片、地图容器、装饰元素

### 8.3 图表色板

单独维护：

- 分类色板
- 顺序色板
- 强调色
- 告警色
- 深色大屏专用色板

### 8.4 地图色板

单独维护：

- 底图暗色
- 区域填充
- 高亮区域
- 飞线
- 点位
- 热力映射

---

## 9. V1 不建议引入的东西

当前版本不建议引入：

- 同时维护两套图表主库
- 一开始就接很多地图服务商
- 为 AI 生成能力提前设计复杂图表 DSL
- 为未来数据库直连提前设计超复杂数据建模
- 为未来多人协作提前设计复杂权限系统

---

## 10. 最终拍板建议

如果现在需要给团队一个明确结论，建议直接这样定：

- 框架：`Next.js + React + TypeScript + pnpm`
- UI：`shadcn/ui + Radix + Tailwind`
- 表单：`react-hook-form + zod`
- 图表：`VChart`
- 国内 3D 地图：`高德 JSAPI 2.0 + Loca`
- 地图增强预留：`AntV L7`
- 编辑器交互：`dnd-kit + react-rnd`
- 数据导入：`xlsx`

这套组合最符合当前版本的目标：

- 国内优先
- 视觉优先
- V1 可落地
- 后续还能继续扩展
