## 2. GoView 公开资料能明确证实的内容

下面这些不是我推测，而是公开资料中能明确确认的。

### 2.1 GoView 的能力层

GoView 官方文档和 README 明确写了它提供：

- 项目控制
- 全局控制
- 组件基础配置
- 动画配置
- 请求配置
- 事件配置

这说明 GoView 的编辑器不是“单纯组件拖拽器”，而是具备完整的页面、组件、数据和交互层。

### 2.2 GoView 的主题能力

公开资料明确写了：

- 支持明/暗切换
- 支持主题色变更

说明它有整体主题体系，而不是每个组件单独拼样式。

### 2.3 GoView 的组件池

GoView README 公开列出了大量已完成组件，能明确确认包括：

#### 图表类

- 柱状图
- 横向柱状图
- 折线图
- 单/多折线面积图
- 饼图
- 环形图
- 水球图
- 雷达图
- 散点图
- 对数回归散点图
- 热力图
- 漏斗图
- 中国地图
- 高德地图
- 多种进度图

#### 信息类

- 文字
- 渐变文字
- 词云
- 嵌套网页
- 图片
- 视频

#### 列表类

- 滚动排名列表
- 滚动表格

#### 小组件

- 边框装饰 01~13
- 装饰组件 01~05
- 数字翻牌
- 通用时间
- 数字计数
- 倒计时
- 时钟

### 2.4 GoView 的技术栈

公开资料明确写了：

- Vue3
- TypeScript
- NaiveUI
- ECharts
- Pinia
- 后续整合 VChart

这说明它不是“纯静态 demo”，而是有明确工程化结构的编辑器。

### 2.5 GoView 的真实使用问题

Issues 列表里能看到一些真实问题，这些对我们反而有启发：

- `mac 多选功能无效`
- `编辑页面时，水球图调整大小问题`
- `动态请求数据后图表样式配置串掉`
- `基础分页表格绑定下拉框点击事件无效`
- `地图 dataset 编辑移除部分配置，页面还保留旧配置`

这些问题说明：

1. 多选
2. 缩放
3. 动态请求与样式解耦
4. 表格事件
5. 地图数据与配置一致性

都是大屏编辑器非常真实、而且容易踩坑的核心问题。

## 3. DataV 官方文档能明确证实的内容

### 3.1 右侧面板不是一坨参数

DataV 官方文档明确把组件配置分成：

- 样式
- 数据源
- 高级

这不是我推测，是官方文档直接写的。

这说明成熟系统右侧面板的组织方式，本来就应该是**按任务分层**，而不是全丢到同一个滚动面板里。

### 3.2 样式里有通用属性

DataV 文档明确提到样式里有这些通用项：

- 组件宽高
- 位置
- 旋转
- 不透明度

这说明：

- 通用组件基础能力要统一
- 不同组件类型的特殊配置是在通用项之上加的

### 3.3 数据源不是“只选一个 dataset”

DataV 文档明确说明了数据源配置包括：

- 数据源类型
- 可视化编辑 / 代码编辑
- 数据映射
- 全屏编辑数据
- 复制数据

这对 JaminView 很关键，说明成熟系统的数据层至少要解决：

- 数据从哪里来
- 怎么看
- 怎么编辑
- 怎么映射到组件

### 3.4 颜色不是一个简单文本框

DataV 的颜色配置是一个独立配置项类型。

这说明：

- 颜色编辑本来就应该被视为正式控件
- 不能继续把颜色当普通字符串字段

### 3.5 地图样式是细粒度配置

DataV 地图相关文档明确能看到很多细项配置：

- 大小
- 颜色
- 透明度
- 图层视觉表达

这说明地图组件不该只给一个“主题”下拉框，而应该开放更细粒度的视觉能力。

## 4. 通过研究确认的产品结论

这部分是基于上面的公开证据，再结合编辑器产品经验，得出的较稳结论。

### 4.1 JaminView 不能再做“通用大表单”

证据来源：

- DataV 右侧明确分层
- GoView 明确有组件基础配置 / 请求配置 / 事件配置

所以 JaminView 必须：

- 页面级、组件级、数据级分层
- 不同组件拥有不同面板
- 右侧面板按任务组织

### 4.2 JaminView 不能只做静态样子货

证据来源：

- GoView 公开列出请求配置、事件配置、数据过滤
- GoView issues 里也能看到动态请求、配置一致性问题

所以 JaminView 的目标不能停在：

- 只摆几个图表
- 只改几个颜色

而是要把这些正式纳入范围：

- 数据系统
- 请求配置
- 数据过滤
- 高级事件

### 4.3 JaminView 的组件池必须扩充

证据来源：

- GoView 的组件池很明确且很丰富

所以 JaminView 不能长期停留在：

- 折线图
- 柱状图
- 饼图
- 地图
- 表格

至少还要补上：

- 横向柱状图
- 数字翻牌
- 边框装饰
- 装饰组件
- 时间类组件

### 4.4 JaminView 应该吸收“能力结构”，不是照抄字段

这是一个重要边界。

能证实的是：

- GoView / DataV 都强调分层
- 都有丰富组件
- 都有数据与交互能力

但不能直接说：

- 它们每个字段我们都要一模一样复制

所以 JaminView 要做的是：

- 复刻功能结构
- 重做适合我们产品的面板组织和视觉体系

## 5. 对 JaminView 的直接要求

基于这轮研究，JaminView 编辑器后续必须满足：

### 5.1 系统层

- 工作台
- 页面级配置
- 组件级配置
- 数据级配置
- 请求配置（至少预留）
- 数据过滤（至少轻量）
- 高级事件（至少预留）

### 5.2 组件层

- 柱状图
- 横向柱状图
- 折线图
- 面积图
- 饼图 / 环图
- 文字
- 图片
- 滚动排名列表
- 滚动表格
- 边框装饰
- 装饰组件
- 数字翻牌

### 5.3 面板层

右侧必须变成：

- 页面
- 组件
  - 内容
  - 数据
  - 样式
  - 高级

而不是：

- 一长串混合字段

## 6. 这一轮研究后的最终判断

### 6.1 之前的问题在哪里

之前的问题不是“方向错”，而是：

- 证据不够硬
- 对 GoView / DataV 的学习还不够深
- 很多结论停留在产品经验层，而不是证据层

### 6.2 现在能更稳地说什么

现在我可以更稳地说：

- GoView / DataV 都不是“参数堆”
- 它们都有明显的分层结构
- GoView 的组件池明显比我们现在丰富很多
- GoView 公开资料支持我们把请求配置、数据过滤、高级事件纳入正式范围
- DataV 官方文档支持我们把右侧面板重构成任务分层，而不是当前这种大表单

### 6.3 现在该怎么做

现在最正确的下一步不是继续零碎实现，而是：

1. 以这份研究笔记为依据
2. 重写 JaminView 的编辑器模块化规划
3. 按模块逐个实施

## 7. 主参考决策

为了避免后续继续在 `GoView / DataV` 之间摇摆，当前正式决策如下：

### 7.1 编辑器结构主参考：DataV

原因：

- 官方文档更完整
- 右侧面板分层更清楚
- `样式 / 数据源 / 高级` 这种结构更适合直接转成 JaminView 的编辑器模型
- 对颜色、数据源、地图图层、配置项的公开描述更细

所以后续这些内容，以 DataV 为主参考：

- 右侧面板结构
- 数据源配置结构
- 样式配置结构
- 地图与图表配置的组织方式
- 配置控件的产品化表达

### 7.2 组件池主参考：GoView

原因：

- GoView 更适合用来确认“一个大屏编辑器应该具备哪些组件”
- 它的组件池公开可见，而且更接近国内大屏交付场景

所以后续这些内容，以 GoView 为主参考：

- 组件种类边界
- 大屏装饰组件
- 边框装饰、数字翻牌、时间类组件
- 滚动排名列表、滚动表格等大屏表达组件

### 7.3 最终落地方式

JaminView 后续的编辑器设计采用：

- `DataV 负责结构`
- `GoView 负责组件池`

而不是继续：

- 两边都学一点
- 最后又回到我们自己的旧大表单
