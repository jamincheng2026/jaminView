# JaminView 页面与功能说明

这份文档用来回答一个很实际的问题：

- 现在页面上看到的这些功能，到底是干嘛的？
- 哪些是第一版真正要做的？
- 哪些只是当前设计稿里的占位和演示内容？

它不是 PRD，也不是设计稿说明，而是一份给产品、设计、前端都能快速看懂的“功能翻译文档”。

---

## 1. 先讲一句人话

`JaminView V1` 的核心不是做一个“万能低代码平台”，而是让用户用最短路径完成这件事：

`选模板 -> 创建项目 -> 导入数据 -> 调整页面 -> 预览 -> 发布展示`

所以我们现在所有页面和功能，都应该围绕这条主线理解。

---

## 2. 当前页面各自是干嘛的

## 2.1 首页

### 页面目标

- 让用户明白 JaminView 是什么
- 让用户知道第一版能做什么
- 引导用户进入产品，而不是停留在介绍页

### 页面里的功能分别是什么

- 顶部导航
  - `产品功能 / 模板库 / 登录 / 立即体验`
  - 作用：产品级导航和入口，不承担复杂业务逻辑

- Hero 主视觉
  - 作用：表达“这是一个做数据大屏的平台”
  - 不是编辑器功能本身，而是营销展示

- 核心能力区
  - `模板起步 / 数据导入 / 可视化编辑 / 发布展示`
  - 作用：告诉用户第一版闭环有哪些环节

- 模板展示区
  - 作用：告诉用户“不是从空白开始”，而是从现成模板起步

- 工作流区
  - 作用：把产品使用过程讲清楚

### 当前哪些是 mock / 占位

- 模板预览图
- Hero 中的大屏或编辑器画面
- 首页所有 CTA 目前只是站内跳转

### 这页本质上不是干嘛的

- 不是正式工作台
- 不是编辑器
- 不是文档页

---

## 2.2 项目列表页 `/projects`

### 页面目标

- 承接登录后的工作台入口
- 展示用户已有项目
- 提供“新建项目”的主入口

### 页面里的功能分别是什么

- 顶部导航
  - `项目 / 资源 / 分析 / 设置`
  - 作用：后续工作台多模块入口
  - 第一版真正有意义的是：`项目`

- 左侧边栏
  - `控制台 / 团队模板 / 最近项目 / 已归档 / 回收站`
  - 作用：组织项目和资产
  - 第一版真正优先要用到的是：
    - 控制台
    - 团队模板
    - 已归档

- 搜索框
  - 作用：查找已有项目
  - 当前是 mock，没有真实搜索逻辑

- 新建项目
  - 作用：进入“创建项目 / 选模板”
  - 这是第一版最重要的按钮之一

- 项目卡片
  - 显示：
    - 缩略图
    - 项目名称
    - 简述
    - 更新时间
    - 状态
  - 作用：告诉用户当前有哪些项目，以及项目是否仍在使用

- 底部模板区
  - `从模板开始`
  - 作用：第二入口，帮助用户快速开始

### 当前哪些是 mock / 占位

- 所有项目卡数据
- 项目缩略图
- 状态标签
- 协作者头像

### 这页真正回答的问题

- 我有哪些项目？
- 我要从哪里新建？
- 哪些项目还在继续做？

---

## 2.3 选择模板页 `/projects/new`

### 页面目标

- 让用户不是从空白开始
- 通过模板进入编辑器

### 页面里的功能分别是什么

- 项目名称输入框
  - 作用：创建项目时先给项目命名
  - 这是项目实体的最早一步

- 放弃
  - 作用：退出本次创建流程

- 开始搭建
  - 作用：用当前名称 + 当前选择模板创建项目
  - 第一版里它应该是“确认创建”的主动作

- 左侧模板分类
  - `全部 / 运营 / 销售 / 智慧城市 / 工业物联`
  - 作用：帮助用户按场景筛模板

- 模板卡片
  - 每张卡里包含：
    - 模板缩略图
    - 行业
    - 复杂度
    - 图表数量
    - 数据源类型
  - 作用：帮助用户判断这个模板适不适合自己

- 使用模板
  - 作用：直接选中某个模板，作为创建项目的基础

### 当前哪些是 mock / 占位

- 模板缩略图
- 模板元数据
- 分类数量
- 搜索模板

### 这页真正回答的问题

- 我先用哪一套模板开始？
- 这套模板适不适合我的场景？
- 项目一开始叫什么名字？

---

## 2.4 数据导入页 `/data-import`

### 页面目标

- 把用户自己的数据带进系统
- 让模板里的假数据，替换成真实业务数据

### 这页为什么重要

因为“模板只是壳子”，真正能让用户觉得产品有价值的，是：

- 他能把自己的数据放进来
- 页面内容开始“像自己的项目”

### 页面里的功能分别是什么

- 数据集名称
  - 作用：给这次导入的数据起一个名字
  - 方便后续多个组件绑定同一个数据集

- 存储位置
  - 作用：决定数据归属到哪个工作区或目录
  - 第一版可以先弱化，只保留一个默认位置

- 上传区域
  - 作用：把 CSV / Excel 文件导入进来
  - 第一版核心支持：
    - `.csv`
    - `.xlsx`

- 文件状态条
  - 显示：
    - 文件名
    - 文件大小
    - 行数
    - 列数
    - 当前是否可导入
  - 作用：告诉用户“系统已经识别到这个文件了”

- 校验提醒
  - 作用：告诉用户数据里可能有问题，但不一定阻塞导入
  - 比如：
    - 日期格式不一致
    - 空值
    - 类型不统一

- 数据映射预览
  - 显示：
    - 字段名
    - 推断出的字段类型
    - 示例值
  - 作用：让用户确认系统识别的数据是否合理

- 高级映射设置
  - 作用：未来扩展字段映射、类型转换、列重命名
  - 第一版可先不做复杂逻辑

- 取消 / 导入
  - `取消`
    - 退出本次导入
  - `导入`
    - 确认将数据集保存到当前项目

- 数据已加密并保持私有
  - 作用：信任提示
  - 第一版更多是产品表达，不是已经实现了完整权限系统

### 当前哪些是 mock / 占位

- 文件内容
- 字段推断结果
- 校验提醒
- 导入结果

### 这页真正回答的问题

- 我能不能把自己的表格放进来？
- 系统认出来的数据对不对？
- 导入之后，组件能不能开始绑定这些字段？

## 2.5 编辑器页

### 页面目标

- 真正修改大屏内容
- 完成页面布局、图表、地图和数据绑定

### 页面里的功能分别是什么

- 顶部栏
  - `返回 / 项目名 / 保存状态 / 预览 / 发布`
  - 作用：管理编辑流程，不是内容编辑本身

- 左侧栏
  - `组件库 / 图层 / 数据源 / 模板`
  - 作用：决定“加什么”、“选什么”、“改什么”

- 中间画布
  - 作用：真正的大屏编辑区域
  - 第一版画布基准是：
    - `1920 x 1080`

- 右侧属性区
  - `基础属性 / 样式 / 数据绑定 / 图表配置 / 地图配置`
  - 作用：修改当前选中组件

### 这页真正回答的问题

- 这张大屏长什么样？
- 每个组件放哪里？
- 它绑定什么数据？
- 样式和图表表现怎么调？

---

## 2.6 预览页

### 页面目标

- 让用户在“接近最终展示”的状态下检查页面

### 页面里的功能分别是什么

- 更干净的顶部栏
  - 返回 / 全屏 / 发布
  - 作用：弱化编辑痕迹，强调观看效果

- 中间大屏内容
  - 作用：查看实际展示效果

### 这页真正回答的问题

- 这张大屏投到展示环境里会是什么效果？
- 字体、间距、图表、地图会不会出问题？

---

## 2.7 发布成功页

### 页面目标

- 给用户明确反馈：发布成功了
- 提供后续动作入口

### 页面里的功能分别是什么

- 发布成功状态
  - 作用：确认本次操作完成

- 展示链接
  - 作用：可复制、可访问

- 查看展示页
  - 作用：跳到最终只读展示页

- 返回编辑器
  - 作用：继续改草稿

### 这页真正回答的问题

- 发布是不是成功了？
- 我现在能把这个展示给谁看？

---

- 让用户浏览全部模板

### 页面目标

- 把用户送进工作台

### 页面里的功能分别是什么

  - 作用：告诉用户这是哪个产品

- 右侧登录表单

## 3. 现在很多功能为什么会让人看不懂

因为现在有 3 层东西叠在一起：

1. 设计稿里的“产品感表达”
2. 第一版真正会做的功能
3. 当前还没接真实逻辑的 mock 占位

所以你会感觉：

- 页面上东西很多
- 但不知道哪些真的会做
- 也不知道每个控件背后的业务动作是什么

这是正常的，不是你看不懂，而是我们还没把“视觉稿”和“真实产品功能”完全翻译到同一层。

---

## 4. 第一版真正应该优先理解的功能

如果只抓第一版最关键的，不要被页面上所有元素吓到，真正重要的是这 6 件事：

1. 从模板创建项目
2. 导入静态数据
3. 在编辑器里调整内容
4. 预览展示效果
5. 发布为只读页面
6. 在项目列表里管理已有项目

其他很多元素目前都可以理解成：

- 辅助表达
- 后续扩展位
- 或设计稿里的产品质感组成部分

---

## 5. 我建议你现在怎么理解这个产品

不要把它理解成“功能很多的复杂平台”，先把它理解成：

> 一个帮用户更快做出并发布数据大屏的模板化工具

这样再看各页面，就会清楚很多：

- 首页：介绍这个工具
- 项目列表：看你做过哪些东西
- 选模板：决定从哪个基础开始
- 数据导入：把你的数据换进去
- 编辑器：改页面
- 预览：看结果
- 发布成功：拿链接去展示

---

## 6. 下一步建议

现在最值得继续补的不是更多页面，而是：

1. 把每个页面的“主任务”固定住
2. 把每个页面中哪些元素是第一版要做的，哪些是后置的，标清楚
3. 再继续推进下一页开发

如果需要，下一步可以继续补一份：

- `哪些功能是 P0 必做`
- `哪些功能现在只是占位`
- `每页主按钮点下去之后应该发生什么`
